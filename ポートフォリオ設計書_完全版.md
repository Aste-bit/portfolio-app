# ポートフォリオ設計書 — 完全版

> **このファイルの目的**: Claude Codeにこのファイルを渡すだけで、最高品質のポートフォリオサイトを実装できるようにするための設計書。デザイン・コンテンツ・技術仕様をすべて含む。

---

## コンセプト

**「静かにすごい」**

自慢しない。アピールしない。ただ丁寧に、やってきたことを見せる。見た人が勝手に「この人すごいな」と感じるポートフォリオ。高級ブランドがロゴを目立たせないのと同じ — 品質そのもので語る。

ポートフォリオ自体のデザインの完成度と、書かれている内容の深さが、言葉で説明するより雄弁に実力を証明する。

トーン: 丁寧な自己紹介。友人に「最近こういうの作ったんだよね」と話すような自然さ。

## デザインの方針

### やること
- 余白をたっぷり取る（情報密度を下げることで品質感が出る）
- タイポグラフィで魅せる（フォントサイズの差でリズムを作る）
- アニメーションは控えめに（ふわっと出る程度。派手な演出は不要）
- セクションごとにレイアウトを変える（カード→テキスト→画像→テキストのリズム）
- スクリーンショット枠を設ける（画像がなくてもレイアウトが成立する設計に）

### やらないこと
- 数字を大きく出して「2プロジェクト！200人！」とアピールしない
- 「実績あり」「対応可能」のバッジを付けない
- Before→Afterの比較レイアウトにしない（営業資料っぽくなる）
- 「すごいでしょ？」と感じさせるコピーを書かない
- アニメーションを派手にしない（技術デモではない）

**一般向けの説明文:**
```
ToC向け情報商材サービス「スキルプラス」の顧客対応を管理するために作ったシステムです。

もともとは21項目あるスプレッドシートとDiscordで運用していたのですが、
```

**技術版の説明文:**

```
ToC向け情報商材サービス「スキルプラス」の顧客対応（抗弁書・弁護士・消費者センター案件）を

インライン編集は楽観的更新パターンで、UIを即座に反映しAPI失敗時にロールバック。
GASのonEditで双方向自動移動（ステータス変更・種別変更）を実装しています。
```

**機能タグ（プロジェクトの下に小さく並べる）:**

一般向け:

技術版:

**設計で考えたこと（技術版のみ・アコーディオン）:**

1. **認証方式**: サービスアカウント → 組織ポリシーでブロック → OAuth2に変更。結果的にユーザー識別が可能に。
2. **ステータス設計**: 初期4段階 → 「保留中」が曖昧 → 「先方待ち」「要対応」に分割して5段階に。ステータスだけで待ちの方向がわかるように。
3. **シート構成**: 1シート → チーム要望で3シート分割。ウェブアプリには「すべて」タブで統合ビューを追加し、両方のニーズに対応。
4. **改善サイクル**: v1リリース後にチームFBでv2改修（3シート分割+ステータス再設計）、さらにUI品質改善21項目を実施。

#### プロジェクト2

同じ構造で、末端ゴール通知システムを記載。

**一般向けの説明文:**
```
自社のゴール管理ツール「Addness」を使って、200名の社員に毎朝
「今日集中すべきゴール」を通知するシステムです。

ゴールには階層構造があり、一番下にある具体的なゴール（末端ゴール）が
日々フォーカスすべきタスクになります。このデータを自動で取得して整理し、
毎朝Google Chatで届くようにしました。

さらに「ゴールが設定されていない人」「期限が過ぎている人」「ゴールの
粒度が粗い人」を自動で検出する仕組みも入れています。管理職が200人の状況を
毎朝把握できるようになることが目標で、現在データ取得の基盤部分が完成しています。
```

**技術版の説明文:**
```
自社ゴール管理SaaS「Addness」のobjectives APIからゴール階層データを取得し、
末端ゴール（子ゴール0件）を判定して毎朝通知するシステム。4コンポーネント構成
（Chrome拡張 + GASバックエンド + Google Chat + GAS Web App）で設計しました。

Chrome拡張v10でAddness APIから全メンバーのゴールを取得（parentObjectiveIdで
ツリー構築）。GAS doPostでRAWデータを受け取り、1分後の一時トリガーで
processGoalTreeを実行する分離設計にしています（GAS 6分制限対策）。

異常検知は4基準（ゴール未設定/粒度が粗い/期限超過・停滞/分解漏れ）を
すべてルールベースで判定。当初5基準でしたが、APIにprogressフィールドが
存在しないことが判明し「進捗ペース異常」を削除しました。

スプレッドシートは既存のスプシ③と共用せず、スプシ④として新規独立。
既存GASとのdoPost衝突やトリガー混在を回避する設計です。
```

**機能タグ:**

一般向け:
- 毎朝自動通知 / 問題の自動検出 / 5画面ダッシュボード / 200人規模対応 / 完全無料 / ログイン不要

技術版:
- Chrome Extension (Manifest V3) / GAS doPost + Trigger分離 / 4基準ルールベース異常検知 / Google Chat Webhook / GAS Web App + React SPA / Session.getActiveUser() 認証 / 7シート構成 / 50人バッチ Token Refresh

**設計で考えたこと（技術版のみ）:**

1. **独立スプシ**: 既存スプシ③と共用 → doPost衝突リスク → スプシ④を新規作成。壊れても既存システムに影響ゼロ。
2. **スケール対応**: 35人想定 → 全員通知(200人)に拡大 → 部署別Webhook + バッチ処理 + doPost/processGoalTree分離。
3. **異常検知基準**: 5基準 → APIにprogress無し → 4基準に。将来preview APIで復活可能な設計。
4. **通知方式**: Google Chat（無料・Webhook一発）を推奨。LINEは月5,000円だが後追い対応できる設計に。

**スクリーンショットのプレースホルダー:**

各プロジェクトにスクリーンショット枠を1つ配置。画像がない場合は薄いグレー背景（bg-stone-100 dark:bg-stone-800）に「Screenshot」テキストを中央配置。角丸（rounded-xl）、影（shadow-sm）。アスペクト比は16:9。画像ファイルは `/public/screenshots/project1.png` と `/public/screenshots/project2.png` を参照するようにし、ファイルがなくてもエラーにならないようにする。

### 開発スタイル

ここも押し付けがましくなく、自然に語る。

```
┌──────────────────────────────────────────┐
│                                          │
│  つくりかた                               │
│                                          │
│  [説明文 — 2段落]                          │
│                                          │
│  ── ── ── ── ── ── ── ── ── ── ──       │
│                                          │
│  整理する → 設計する → 作る → 届ける → 直す │
│   💬          📝        ⚡      🚀       🔄  │
│                                          │
│  ★ 技術版のみ:                            │
│  Phase方式の詳細 [展開/折りたたみ]          │
│                                          │
└──────────────────────────────────────────┘
```

**一般向け説明文:**
```
自分ではコードを書きません。

業務の課題をじっくり聞いて、最適な仕組みを設計するところに集中し、
プログラミングはAI（Claude Code）に任せています。
「コードが書けなくても、業務をよく理解していれば良いシステムは作れる」
というのが、2つのプロジェクトを通じて実感していることです。
```

**技術版説明文:**
```
Claudeチャットで業務分析・要件定義・アーキテクチャ設計を行い、
設計書をClaude Codeに渡して実装させるAI駆動開発スタイルです。

設計書には型定義・API仕様・画面設計・ディレクトリ構成を漏れなく記述し、
実装の忠実度を担保しています。改修時は変更箇所だけをまとめた
差分プロンプトを作成。Phase方式で各フェーズ完了後に次に進む
段階的な開発フローを採用しています。
```

**プロセスフロー:**

## アニメーション仕様

**原則: 控えめに、上品に。**

- 全セクション共通: スクロールで画面に入った時に `opacity: 0→1` + `y: 20→0`（微かに上がる程度）
- duration: 0.7s, ease: "easeOut"
- `viewport: { once: true, margin: "-50px" }`
- カード群は stagger: 0.05s（速めのstaggerでさりげなく）
- ホバーエフェクト: ボーダー色がアクセントに変わる + shadow-sm→shadow-md。scaleは使わない（やりすぎ感が出る）
- モード切り替え時: AnimatePresenceで opacity 0→1（0.3s）。layoutアニメーションで高さの変化を滑らかに。
- ページ読み込み時: イントロの名前だけが1s fadeIn。他は通常速度。

**使わないアニメーション:**
- カウントアップ（数字を0から増やすやつ）
- バウンス
- スライドイン（左右から飛んでくるやつ）
- パーティクル、グラデーションアニメーション等の装飾的なもの

```jsx
// 共通の控えめフェードイン
const fadeIn = {
  initial: { opacity: 0, y: 20 },
  whileInView: { opacity: 1, y: 0 },
  viewport: { once: true, margin: "-50px" },
  transition: { duration: 0.7, ease: "easeOut" }
};

// stagger
const stagger = {
  initial: {},
  whileInView: { transition: { staggerChildren: 0.05 } },
  viewport: { once: true }
};
```

## Claude Code向け実装指示

```
このフォルダのVite + Reactプロジェクトで、ポートフォリオサイトを実装してください。

設計書「ポートフォリオ設計書_完全版.md」に全仕様が記載されています。

追加インストール:
npm install framer-motion lucide-react

コンセプト:
「静かにすごい」— 自慢しない、アピールしない、丁寧に見せる。
余白をたっぷり使い、読み物のようなゆったりした体験にする。
アニメーションは控えめ（fadeIn + 微かなスライドアップのみ）。

守ること:
- 設計書のカラーシステム（Stone系ウォームグレー）を忠実に
- タイポグラフィの指定を忠実に
- テキスト内容は設計書のコンテンツデータをそのまま使う
- 余白は多めに。詰め込まない
- max-w-3xl（狭め）で読み物感を出す
- スクリーンショットは /public/screenshots/ から読み込み、画像がなくてもエラーにならないプレースホルダーを表示

自由にしてよいこと:
- コンポーネントの分割粒度
- Tailwindクラスの具体的な書き方
- アニメーションのduration微調整
- アコーディオンの実装方法

各ステップごとに npm run dev で動作確認しながら進めてください。
Introセクションから順番に作り、各セクション完成ごとに見た目を確認してください。
完成したら npm run build でビルドが通ることを確認してください。
```
